# 章データ作成用プロンプト雛形(ChatGPT/Claude チャットにコピペで使う)

TASの補助作業で、章データ(chapter_XX.json)の断片をチャットAIに書かせるための雛形。
API連携はしない。**プロンプトを貼る → 返ってきたJSON断片を章データに貼る → 最後に必ずgemma4のテキストテスト(mock2中継サーバー経由)で確認**、が確定フロー。

- スキーマの正: `docs/DATA_SCHEMA_v0.1.md`(食い違ったらそちらが正しい)
- 文体の正: `public/data/campaign.json` の `style`(雛形に転記済み。campaign.jsonを変えたらここも更新すること)

使い方: `【】` の箇所を埋めて全文コピペ。共通ヘッダは毎回先頭に付ける。

---

## 共通ヘッダ(全雛形の先頭に付ける)

```text
あなたはソロTRPG「Terminus」の章データを書くシナリオライターだ。以下のルールを厳守せよ。

# 世界観
舞台は電気も内燃機関もない前近代の世界。現代の機器や語彙(懐中電灯・電灯・モーター・エンジン・電池・メートル)を出してはならない。

# 文体(プレイヤーに見える全テキスト共通)
- 日本語の「である調」(だ・である)。ですます調は禁止
- 地の文は3文以内、合計80字以内。1文は短く、主語と動詞だけのシンプルな形
- 小学校高学年が読んですぐ分かる、やさしい言葉だけを使う。難しい漢語・比喩・抽象語は禁止
- 内面や状況を評価語で語らず、見えたこと・聞こえたことをそのまま短い動詞で書く
- 良い例: 「廊下は暗くて静かだ。足音がよく響く。」
- 悪い例: 「薄明の廊下に、沈黙を湛えた気配が漂っている。」
- 「軌条」など説明なしで通じない言葉は使わず、やさしい言い換え(レール等)を使う

# 出力
- 指定したJSON断片だけをコードブロックで出力する。前置き・解説・補足は書かない
- JSONの値以外の場所(キー名・構造)を変えない。指定のないフィールドを発明しない
```

---

## 雛形1: シーン一式を作る

共通ヘッダに続けて:

```text
# 依頼
新しいシーンを1つ書け。

- 章の題材: 【章のあらすじ・依頼内容を2〜3行で】
- このシーンの役割: 【例: 中盤の探索。◯◯の伏線を2つ置く。敵はいない】
- 直前のシーン: 【名前と終わり方】
- 直後のシーン: 【名前。このシーンから何が繋がるか】

# 出力形式(この構造のJSONだけを出力)
{
  "id": 【番号】,
  "name": "シーン名(短く)",
  "brief": "シーン開始時にプレイヤーへ見せる説明。文体ルール厳守",
  "img": "【背景ファイル名.jpg(仮名でよい)】",
  "goal": "GM向け: このシーンの完了条件を1文で",
  "direction": "GM向けの演出指示。何を匂わせ、何をまだ言わないかを2〜3文で",
  "loot": [],
  "secrets": []
}

secretsとlootは空配列のままでよい(別の雛形で作る)。
```

## 雛形2: secrets(発見項目)を作る

共通ヘッダに続けて:

```text
# 依頼
シーンの発見項目(secrets)を【2】個書け。

- シーンの状況: 【briefを貼る】
- このシーンで明かしてよい真相: 【箇条書き。どこまで言ってよいかの上限も書く】
- まだ明かしてはいけないこと: 【箇条書き】

# 各フィールドの意味(重要)
- entity: 調査対象の名前。プレイヤーの入力補助チップにそのまま出る(短い名詞)
- surface: 未開示でもプレイヤーに見えている表層描写。真相のネタバレ禁止
- text: 開示時にGMが得る確定事実。「※この開示で語ってよいのは◯◯まで」の注記を付けてよい
- playerText: textにGM向け注記を付けた場合、プレイヤーに直接見せる版(注記なし)を書く。不要なら省略
- aliases: プレイヤーが入力しそうな言い換え・略称を3〜6個(例: 「木柵」→「柵」「封鎖」)

# 出力形式(この構造のJSON配列だけを出力)
[
  {
    "id": "s【シーン番号】a",
    "entity": "…",
    "surface": "…",
    "text": "…",
    "aliases": ["…"]
  }
]
```

## 雛形3: 敵を作る

共通ヘッダに続けて:

```text
# 依頼
シーンの敵を1体書け。数値はゲームバランス用の目安に従う。

- シーンの状況: 【briefを貼る】
- 敵のコンセプト: 【例: 縄張りを守る獣。倒さなくても光で追い払える】
- 強さの目安: 【弱め/標準/強め】

# 数値の目安(d20、プレイヤーHP10、与ダメ1が基準)
- hp: 弱め4〜5 / 標準6〜8 / 強め9〜12
- agility: 行動順。プレイヤー6・ガレス7・リディア4より速いか遅いかで決める(1〜10)
- atk: 命中時ダメージ。通常1、強敵のみ2
- defenseDc: この敵への攻撃判定DC。弱め11 / 標準12〜13 / 強め14
- fleeDc: 逃走判定DC。逃げやすい8〜9 / 普通10 / 逃げにくい12
- ambushDc: 奇襲回避DC(奇襲するなら)。10〜13

# 出力形式(この構造のJSONだけを出力。弱点が無いならweaknessごと省略)
{
  "name": "正名",
  "unknownName": "未識別時の表示名(正体を隠すなら。例: 不気味な影)",
  "hp": 0, "maxHp": 0,
  "surface": "未識別段階でも語ってよい見た目。文体ルール厳守",
  "trait": "GM向けの性質(弱点等を含む)",
  "ambush": false,
  "ambushDc": 12,
  "ambushTrigger": "奇襲が発生する条件の文章",
  "agility": 5, "atk": 1, "defenseDc": 12, "fleeDc": 10,
  "weakness": { "triggers": ["宣言に含まれたら発動する語"], "effect": "flee", "text": "発動時の描写1文" }
}
```

## 雛形4: 同行者のセリフ集を作る(campaign.json向け)

共通ヘッダに続けて:

```text
# 依頼
同行者のセリフを書け。

- キャラクター: 【名前と人格。campaign.jsonのpersonaを貼る】
- 口調の見本: 【既存セリフを2〜3個貼る】
- 書くもの: 【例: battleEnd(win/fled/repelledを各3本) / quirksのmutter5本】

# ルール
- 1本は40字以内。かぎかっこ「」は付けない(システムが付ける)
- 状況説明はセリフに含めない。そのキャラが言いそうな一言だけ

# 出力形式(この構造のJSONだけを出力)
{
  "battleEnd": {
    "win": ["…", "…", "…"],
    "fled": ["…", "…", "…"],
    "repelled": ["…", "…", "…"]
  }
}
```

## 貼り込み後のチェックリスト

1. 正名(敵名・loot名)を `campaign.json` の `entities` 台帳に登録したか
2. `npm run check:assets` が通るか(imgを仮名にした場合はassets.jsonにpendingで登録)
3. TAS側 `validate.cjs` が通るか
4. **gemma4:e4bのテキストテスト**で、direction・secretsが本番モデルでも守られるか(強いモデルで通る指示が小さいモデルで守られるとは限らない)
