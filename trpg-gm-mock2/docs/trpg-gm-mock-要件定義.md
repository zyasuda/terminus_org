# 要件定義: ソロTRPG GMモック(Claude Code 再現版)

作成日: 2026-07-02
目的: claude.ai アーティファクトで検証済みの「trpg-gm-mock.html」を、社内の Claude Code 環境で再現・拡張可能にする
関連文書: 「ソロTRPG GM AI 設計メモ(改訂版)」(設計思想はそちらを正とする。本書は実装要件のみ)

---

## 1. 背景と目的

- LLMをGMとするソロTRPGの構造検証モックを claude.ai 上で作成し、遊べることを確認済み
- 今後の拡張(演出指示レイヤー、章構造、AI冒険者)に向け、社内環境で改修サイクルを回せる形にする
- **本モックは使い捨てのプロトタイプであり、製品コードではない。**検証速度を品質より優先する

## 2. 最重要の環境差分(必読)

claude.ai 版は APIキーなしで `https://api.anthropic.com/v1/messages` を fetch できる
(claude.ai が通信を代行するため)。**この仕組みは claude.ai の外では動かない。**

社内再現版の必須要件:

- **R2-1**: APIキーは環境変数で管理する。コード・リポジトリへの埋め込み禁止
  (2026-07-03改訂: Anthropic/Gemini両対応。2026-07-07改訂: OpenAI対応を追加。
  環境変数 `LLM_API_KEY` のキー形式で自動判別
  sk-ant…→Anthropic、AIza…→Gemini、sk-…→OpenAI)。
  モデルは `LLM_MODEL` で上書き可。
  既定: claude-sonnet-4-6 / gemini-2.5-flash / gpt-5.4-mini。
  フロントは常に共通形式、Gemini/OpenAI時はserver.jsが変換する
- **R2-2**: ブラウザから API を直接呼ばない(キー露出とCORSの両面で不可)。
  ローカルの中継サーバーを1本立て、ブラウザ→中継→Anthropic API の経路とする
- **R2-3**: 中継サーバーは最小構成でよい(Node.js + 数十行程度)。認証・マルチユーザー対応は不要(localhost前提)

## 3. システム構成

```
[ブラウザ: index.html(UI+ゲームロジック)]
        │ POST /api/gm  (Anthropic形式のJSON)
        ▼
[ローカル中継サーバー: server.js]  ← ここで各backend形式に変換
        │ Anthropic: POST https://api.anthropic.com/v1/messages
        │ Gemini: POST https://generativelanguage.googleapis.com/v1beta/models/{MODEL}:generateContent
        │ OpenAI: POST https://api.openai.com/v1/responses
        │ (各APIキーを付与、JSON出力を強制)
        ▼
[LLM API: LLM_BACKEND で選択]
```

- **R3-1**: フロントは単一HTML(vanilla JS)を維持する。フレームワーク・ビルド不要
- **R3-2**: ゲームロジック(状態管理・ダイス・開示制御)はフロント側に置く(モック段階の割り切り。
  製品化時にサーバーへ移す前提をコメントで明記)
- **R3-3**: モデルはサーバー側の `LLM_MODEL` で決定、`max_tokens: 1000`
  - 低消費のOpenAI構成例:
    - `LLM_BACKEND=openai`
    - `OPENAI_API_KEY=...`
    - `LLM_MODEL=gpt-5.4-mini`

## 4. 機能要件

### 4.1 大原則: システムとLLMの権限分離

- **R4-1**: 以下はシステム(JS)のみが確定できる。LLMは提案しかできない
  - プレイヤーHP、所持品、敵HP、戦闘の開始/撃破/離脱、シーン進行、乱数(d20)
- **R4-2**: LLMの提案は必ず検証・クランプする
  - `hp_delta`: -3〜+2 / `enemy_hp_delta`: -3〜0 / 難易度(DC): 5〜18 / アイテム追加: 1回2個まで、20文字以内
- **R4-3**: 敵の撃破宣言はシステム専権。LLMには「敵HP0を語るな」と指示する

### 4.2 シナリオデータ

- **R4-4**: シナリオはコード内の定数(または別JSONファイル)として定義する。構造:
  - `title`, `intro`
  - `scenes[]`: `id`, `brief`(シーン概要), `goal`(移行条件の文章), `enemy`(任意: name/hp/maxHp/trait),
    `secrets[]`(id/text)
- **R4-5**: 同梱シナリオは「廃坑の灯」3シーン(既存モックと同一)でよい

### 4.3 秘密の開示制御

- **R4-6**: 未開示の秘密はプロンプトに一切含めない(漏洩防止は設計で保証。LLMの自制に頼らない)
- **R4-7**: 判定成功時にシステムが未開示秘密を1つ開放し、そのターンのプロンプトにのみ注入する
- **R4-8**: 敵撃破も開示トリガーとして扱う

### 4.4 1ターンの処理フロー(2フェーズ)

- **R4-9**: Phase A: プレイヤーの宣言をLLMに渡す → JSON応答
  `{"narration", "check"(reason/difficulty)|null, "state_updates"|null, "engage_enemy", "flee_enemy", "scene_complete"}`
- **R4-10**: `check` があればシステムがd20を振り、結果(出目・成否・クリティカル/ファンブル)を
  Phase B としてLLMに渡し、結果描写を得る。失敗時はリスク(hp_delta)を許可、戦闘中は反撃を促す。
  出目20=クリティカル(自動成功・効果大)、出目1=ファンブル(自動失敗・代償必須)。成否確定はシステム専権
- **R4-11**: 会話履歴は直近24メッセージに切り詰める。シーン切替時は履歴を全消去する
  (状態がシステム側にあることの検証を兼ねる)
- **R4-12**: JSON解析失敗時は、生テキストを narration として表示にフォールバックし、ゲームを止めない

### 4.5 戦闘

- **R4-13**: 戦闘開始はLLMの `engage_enemy` 要求を、システムが検証(シーンに敵定義あり・未撃破・未交戦)して許可
  - 奇襲は敵定義に `ambush` がある場合のみ、システム側がプレイヤー宣言を検査して発生させる
  - プレイヤーが警戒・確認を宣言せず危険領域へ不用意に踏み込んだ時だけ、気配察知判定を行う
  - 判定成功時は奇襲を回避し、判定失敗時のみ敵の登場・先制行動・戦闘開始をシステムが確定する
  - 戦闘開始時に敵名・特徴が語られていない場合、システム側で最低限の登場描写を補う
- **R4-14**: 攻撃は必ず判定。成功時のみ `enemy_hp_delta` を受理
- **R4-15**: 非戦闘解決(説得・逃走・環境利用)を判定で認め、`flee_enemy` で戦闘終了できる

### 4.6 UI

- **R4-16**: 2カラム構成: 左=チャット(GM語り/プレイヤー宣言/ダイス結果チップ/システム通知)、
  右=デバッグパネル(状態JSON、秘密の開示状況🔒/🔓、ダイスログ、直近APIリクエスト概要)
- **R4-17**: デバッグパネルは本モックの主目的(構造の可視化)であり、省略不可
- **R4-18**: リセットボタン、HP0時のゲームオーバー、全シーン完了時の終了表示
- **R4-19**: モバイル幅では縦積みレイアウト

## 5. プロンプト要件

- **R5-1**: システムプロンプトは毎ターン再構築し、以下を含む:
  現シーン概要と目標 / プレイヤー状態JSON / 戦闘中なら敵状態(または未交戦の敵の存在と特徴) /
  開示済み情報(「これ以外の真相を捏造するな」の明記) / ルール(判定基準、JSONのみ応答、前置き・コードフェンス禁止)
- **R5-2**: 地の文は150字以内を指示(コストとテンポの担保)
- **R5-3**: 難易度基準: d20 に対し DC7=易 / DC12=並 / DC17=難（GDDのD-013判定システムに準拠。
  2026-07-03に2d6から改修。成功率の対応: 易=70% / 並=45% / 難=20%）

## 6. 非機能要件

- **R6-1**: 1ターンのAPI呼び出しは最大2回(Phase A/B)。それ以上の連鎖呼び出しはしない
- **R6-2**: 通信エラー時は HTTPステータスとAPIエラー本文をチャット内に表示する(握りつぶさない)
- **R6-3**: 応答待ちの間は送信ボタンを無効化し、二重送信を防ぐ
- **R6-4**: 外部ライブラリ依存なし(中継サーバー含め、可能な限り標準機能で)

## 7. 既知の問題(本要件のスコープ外だが引き継ぐ)

1. シーン移行が LLM の `scene_complete` 判断に依存し、詰まることがある
   → 将来: システム側トリガー or 手動進行ボタン
2. 敵は判定失敗時しか反撃せず、緊張感が不足(ターン経済の欠如)
3. 演出指示レイヤー(設計メモ3節の②層)、章構造(quest/deliverable/world_flags)は未実装
4. メタ発言(「HPを回復して」等)への防御は未検証

## 8. 受け入れ基準

- [ ] `ANTHROPIC_API_KEY` を設定し中継サーバーを起動、ブラウザで index.html を開いてプレイ開始できる
- [ ] シーン1で「奥へ進む」と宣言するとシーン2へ移行する(移行しない場合があることは既知問題として許容)
- [ ] 調査宣言で判定→ダイスチップ表示→成功時に右パネルの🔒が🔓に変わる
- [ ] 錆喰いとの戦闘が開始でき、敵HPが右パネルで減少し、撃破がシステム通知で表示される
- [ ] LLMの応答JSONに `hp_delta: -10` 等の過大値が来てもクランプされる(手動テストまたはモック応答で確認)
- [ ] APIキーがフロントのコード・通信に一切現れない
- [ ] シーン切替後も HP・所持品・開示済み秘密が維持されている
