# シーン遷移モデル v0.2 叩き台(TAS合意用)

status: **draft — TAS側(Codex)との合意待ち。合意後に DATA_SCHEMA を v0.2 に更新する**

## 何を解決したいか(2026-07-17(10) クロニクルの実害)

現行 v0.1 の遷移は「scenes[] 配列の直列。move宣言で completeRequires を満たしていれば次の要素へ」だけ。方向・行き先の概念がないため:

1. シーン2で「**左側**に進む」(briefでは行き止まり)でも、条件さえ満たせば右の坑道の先=シーン3へ遷移してしまう
2. シーン3(坑道の最奥)で「**奥**に進む」と、次の配列要素=シーン4「村への帰還」へ遷移する。宣言の内容と遷移先の物語的整合が取れない

エンジン側の原則は既存どおり: **遷移の可否と行き先はシステムが決定論で解決し、LLMは関与しない**(docs/RULE_INVENTORY.md A系)。

## 提案: scenes[].exits[]

```jsonc
{
  "id": 2,
  "name": "分かれ道",
  "exits": [
    {
      "id": "right",
      "match": ["右", "柵", "隙間", "奥", "第三坑道"],   // 宣言との照合語(いずれかを含む)
      "to": 3,                                            // 遷移先シーンid。"end"=章の決着
      "requires": { "secretsAny": ["s2a", "s2b"] },       // 満たさない場合は blockedText
      "blockedText": "封鎖の木柵が行く手をふさいでいる。…"
    },
    {
      "id": "left",
      "match": ["左"],
      "to": null,                                          // null = 行き止まり(遷移しない)
      "text": "左の坑道は崩れて行き止まりだ。足元の瓦礫が崩れる音だけが響く。"
    },
    {
      "id": "back",
      "match": ["戻る", "引き返す", "入口"],
      "to": 1                                              // 後戻りもexitとして書ける(v0.2では任意)
    }
  ]
}
```

### 解決ルール(エンジン実装、決定論)

1. move意図(辞書 or LLM分類器)と判定された宣言に対してのみ exits を評価する
2. `match` の照合語を含む exit を探す。複数一致は **配列の先頭優先**
3. どの exit にも一致しない move は **デフォルト exit**(後述)に落とす
4. 一致した exit の `requires` を検証(現行 completeRequires と同じ語彙: `secretsAny` / `secretsAll` / `talkTurnsMin`)。不成立なら `blockedText`
5. `to: null` は遷移せず `text` を定型ナレーション(行き止まり・崖・施錠扉)
6. `to: "end"` は章の決着(現行の最終シーン完了と同じ処理)

### 後方互換(重要)

`exits` が無いシーンは **現行どおり「次の配列要素へ」**。つまり v0.1 のデータはそのまま動く。
`exits` があるシーンでは、`{"id":"default","match":[],"to":<次のid>}` を明示しない限り、
match不一致のmoveは「先頭の exit の requires 検証」ではなく **最初の `to` が数値の exit** をデフォルトとして扱う
(→ ここは要議論。案B: デフォルト無し=不一致moveは「どちらへ? 」と聞き返す定型文)。

## 遷移のタイミング(v0.2 のスコープ宣言)

BORG `Inbox/TAS_条件と遷移モデル再整理_セッション引き継ぎ_2026-07-15` の4概念のうち:

| 概念 | v0.2 | 理由 |
|---|---|---|
| 行動時遷移(move宣言で exit を選ぶ) | **入れる** | 今回の実害の解決に必須 |
| 条件の AND/OR (`secretsAny` / `secretsAll`) | **入れる** | requires の語彙追加だけで済む |
| 自動遷移(条件成立した瞬間に飛ぶ) | 見送り | 使い所が未確定。演出(強制イベント)と衝突する |
| 条件式の入れ子(AND の中に OR 等) | 見送り | secretsAny+secretsAll の併記(全条件AND)で当面足りる |

## completeRequires との関係

- `exits[].requires` が **exit単位の条件**。`completeRequires` は「シーン全体の完了条件」として残す(exit の requires に共通条件を毎回書かなくて済む)
- 評価順: completeRequires(シーン全体)→ exit.match → exit.requires
- 将来 completeRequires を「全exitの共通requires」に統合する案もあるが、v0.2 では両方残す(既存データ互換)

## TAS側への確認事項

1. exits の `match` 照合語は TAS 編集UIでどう入力させるか(secretsのaliasesと同じ自由記入で良いか)
2. デフォルト exit の扱い: 案A「最初の数値toをデフォルト」/ 案B「不一致は聞き返す」どちらにするか → **mock2側の推しは案B**(誤遷移がなくなる。聞き返し文はシーン共通の定型でよい)
3. `to: "end"` の表現(章エピローグとフラグ確定のトリガー)が TAS の章エディタ上で自然に書けるか
4. 後戻り exit(to が前のシーン)を許すと、シーン状態(開示済みsecret・loot)の巻き戻し無しで再訪することになるが、TAS的に問題ないか(mock2は問題ない: 状態はグローバルに持っている)
