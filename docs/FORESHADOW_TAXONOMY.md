## 0. 前提の整理

### 0.1 既存機構の棚卸し（伏線の受け皿になりうるもの）

| 機構 | 出典 | 伏線への使い方 |
|---|---|---|
| 伏線台帳 | CAMPAIGN_01 3.2 | 設置（②層で匂わせ）＋本体（③層）＋回収（開示＋強制注入）の三点セット。状態 planted→revealed→paid_off |
| ②層演出指示 | AI_DESIGN 2節 | 「事実ではなく効果を書く」ト書き。LLMが真相を知らずに匂わせる装置 |
| 強制注入 | AI_DESIGN 4節 | 期限シーン到達時、「この真相を突きつけて終われ」で回収を保証 |
| world_flags | AI_DESIGN 3節 | 章をまたぐ世界状態。次章の描写・NPCの態度を着色 |
| 秘密の分類（装飾/必須） | AI_DESIGN 2節 | 装飾=ダイス開示可、必須=進行で保証 |
| Chronicle全文記録 | D-020 | GMの語りは全文記録される。**プレイ中に実際に発話されたテキスト**が構造化されて残る |

### 0.2 TRPG固有の制約（映画・脚本との根本的な差）

映画の伏線技法をそのまま持ち込めない理由は3つある。以降の全分類で常にこの3つに照らす。

- **制約T1: 回収タイミングを完全制御できない**
  映画は「92分にこのカット」と指定できる。TRPGはプレイヤーが伏線の現場を素通りしたり、
  逆に予定より早く核心に触れたりする。→ 既存の答えが「期限シーン到達時の強制注入」。
  つまり本システムの回収保証は**時刻指定ではなくチェックポイント指定**である。
  この方式で受け止められる技法と、受け止められない技法（後述の反復系・呼応系）がある。
- **制約T2: 台詞の半分は事前に書けない**
  callback（台詞の呼応）の素材の半分はプレイヤーの発話であり、キャンペーン制作時には
  存在しない。→ 伏線台帳が前提とする「制作時に設置を書き込む」モデルの外にある。
  ただし D-020 により GM の語りは全文記録されている——**設置を事後に採取する**道がある。
- **制約T3: プレイヤーキャラのアークは設計できない**
  脚本の主人公アーク（欠点の設置→変化で回収）を人間プレイヤーに適用することは
  エージェンシーの侵害（AI_DESIGN 4節「評価・決断は語らない」と同根）。
  アーク型伏線を張れる対象は **NPC・AI冒険者・世界そのもの**に限られる。

## 1. 技法の列挙 —— Save the Cat! ビートシートから

Blake Snyder の BS2（15ビート）のうち、伏線として機能する構造を抽出する。
ビートシート自体は「配置表」であり、伏線技法はその中に埋め込まれている。

| ビート | 伏線としての機能 | 技法分類（2章） |
|---|---|---|
| Opening Image / Final Image | 冒頭と結末の対の情景。変化の総量を「同じ絵の差分」で見せる | D. ブックエンド |
| Theme Stated（p.5前後） | 物語のテーマを、序盤に**誰かの台詞として**主人公に投げる。主人公はその時点で理解しない。終盤に意味が立ち上がる | C. 台詞の呼応（テーマ型） |
| Set-up（p.1-10） | 主人公の欠点・直すべき点（"six things that need fixing"）を全て提示。後半で一つずつ回収される | E. キャラクターアーク型 |
| Catalyst / Debate | 伏線というより起動装置だが、「Debateで拒んだ理由」が後の弱点として再登場することが多い | E の変種 |
| B Story（p.30） | サブプロット（多くは関係性の物語）がテーマを別角度から照らす。Aストーリーの解決の鍵がBストーリーから来る | F. Bストーリー照射型 |
| Fun and Games | 「前提の約束（Promise of the Premise）」の消化。ここで見せた道具・技・関係が Finale で再利用される | A. モノ・情報型の連鎖 |
| Midpoint ⇄ All Is Lost | 偽りの勝利⇄偽りの敗北の鏡像。Midpointの高揚の中に崩壊の種を埋める | B. 意味の反転型（構造版） |
| Whiff of Death（All Is Lost） | 死の気配の前兆。トーンで死をほのめかす | G. トーンの前兆 |
| Dark Night of the Soul → Break into Three | Bストーリー起源の気づきが解決をもたらす。Theme Stated の回収点 | C＋F の合流 |

Save the Cat! 以外の一般脚本術から補完する技法：

- **チェーホフの銃**（モノ・情報型）——既存の台帳が扱う型
- **レッドヘリング**（偽の伏線）——注意を逸らすための意図的な誤誘導
- **反復モチーフ / ランニングギャグ**——同じ意匠・仕草・言い回しの反復が、最後の一回で意味を変える（Rule of Three）
- **予言・約束型**——「三度目の鐘が鳴るとき」等、明示的な予告。回収は予告の成就（または裏切り）
- **plant and payoff の逆順（事後伏線）**——先に謎の描写を置き、後から説明する。ミステリの基本形

## 2. 技法分類 —— 「何が設置で、何が回収か」

各技法を GM AI の観点で定義する。判定（既存機構で足りるか）は3章のマトリクスに集約。

### A. モノ・情報型（チェーホフの銃）【既存】

- **設置**: シーンの secrets / ②層演出指示で、モノ・痕跡を描写に混ぜる
- **本体**: ③層の秘密（そのモノの真の意味）
- **回収**: 開示イベント。期限シーンで強制注入
- 現行台帳の F1・F3・F4 がこれ。**現機構で完結**。議論の余地は運用（粒度）のみ。

### B. 意味の反転型【既存・要拡張】

- **設置**: 一度**普通に開示された事実**（①層に入った情報）
- **本体**: 同じ事実の別の意味（③層）
- **回収**: 反転の瞬間の演出。「あの作業札は閉じ込めるためではなく、出さないためだった」
- 現行 F2。台帳で扱えるが、現スキーマには「設置＝未開示の匂わせ」の前提が暗黙にある。
  反転型の設置は**開示済み情報**であり、状態遷移が異なる
  （planted の時点で既に revealed 済みの一次意味を持つ）。
- **拡張候補**: 台帳エントリに `first_meaning`（一次意味・開示済）と `true_meaning`（③層）の
  二段を持たせる。回収時の強制注入は「一次意味を引用してから反転させよ」という
  ②層文法になる——反転は**対比の再提示**がないと成立しないため。
- Midpoint⇄All Is Lost の鏡像構造は、これの章スケール版。world_flags と章の起承転結設計
  （CAMPAIGN_01 3章）が既に担っており、追加機構は不要と考える。

### C. 台詞の呼応（callback / Theme Stated 型）

- **設置**: ある台詞が発話される（GM/NPC発 or **プレイヤー発**）
- **本体**: その台詞が再登場すべき文脈の条件（「番人の最期の場面で」等）
- **回収**: 同じ台詞（または変奏）を、意味が変わった文脈で再発話させる
- 二つの下位型に分かれ、機構要件が全く違う：
  - **C-1. 脚本内呼応（Theme Stated 型）**: 設置台詞を制作時に書ける。
    「灯を守るのは、誰の仕事か」（キャンペーンの問い）を1章でマイラかNPCに
    言わせておき、3章の Finale で番人（または プレイヤーへの問いかけ）として返す。
    → 台帳スキーマの `設置` を「②層演出指示」から「**指定台詞の発話指示**」に
    広げれば載る。ただし回収は「この台詞を変奏して使え」という新しい②層文法が要る。
  - **C-2. プレイ内呼応（動的callback）**: プレイヤーの名台詞・印象的なGMの即興句を
    後で返す。制作時に書けない（制約T2）。
    → **新機構が必要**（4章の「動的伏線台帳」案）。ただし土台は既にある：
    D-020 で GM の語りは全文記録されており、D-016 検証では外部AIが全文ログから
    名台詞（「私はもう十分、灯した」）を自力で発掘できた。
    **Chronicleが二次創作用にやっていることを、プレイ中のエンジンが自分に対してやる**
    のがこの機構の本質。

### D. ブックエンド（Opening Image / Final Image）

- **設置**: 章（またはキャンペーン）冒頭の情景描写。構図・モチーフを固定する
- **本体**: 「終幕は冒頭と同じ構図で、差分だけを変えて描く」という対応関係そのもの
- **回収**: 終幕シーンで、冒頭の情景を引用しつつ world_flags の差分を描き込む
- 例：1章 sc1「朽ちたトロッコ軌道が闇へ続く。奥からかすかな金属音」
  → 3章終幕「同じ坑口。軌道は瓦礫の下。金属音はもう聞こえない——あるいは、
  guardian_alive なら、遠くでまだ一つだけ灯りが揺れている」
- 機構的には**秘密ではない**（③層に隠す真相がない）ので、伏線台帳の三点セットに
  馴染まない。これは**シーン定義のペアリング**であり、章スキーマの拡張が自然：
  `opening_image` / `final_image` フィールドを章に持たせ、final_image の②層に
  「opening_image の brief を引用し、world_flags による差分を対比して描け」と書く。
- 回収保証は容易（終幕シーンは必ず到達する＝制約T1に強い）。
  **費用対効果が最も高い未実装技法**の候補。world_flags の「語りは発散」設計と
  直結し、プレイヤーの選択の総量を一枚の絵の差分として可視化できる。

### E. キャラクターアーク型（欠点の設置 → 変化で回収）

- **設置**: キャラクターの欠点・固着・未解決の過去を、序盤の言動として繰り返し見せる
- **本体**: そのキャラクターが変化しうる条件と、変化後の姿（③層）
- **回収**: 条件成立時、以前と**同じ状況で違う行動**を取らせる（変化は差分でしか描けない）
- 制約T3により、対象は**NPC・AI冒険者のみ**。プレイヤーのアークは設計しない
  （プレイヤーが勝手に成長の物語を生きるのは歓迎するが、エンジンは強制しない）。
- 例：マイラのアーク——「値踏みする目」（欠点＝人を損得で見る）を1〜2章で反復し、
  3章で「損得抜きに村人（または番人）を庇う／あるいは最後まで損得で動く」の分岐を
  world_flags（heartstone_choice 等）に応じて演じ分ける。
- 機構的には三層知識モデルの**時間軸拡張**である：現行の②層は「このシーンでの
  見せ方」を書くが、アークには「章ごとに②層が段階遷移する」仕掛けが要る。
  キャスト定義の②層を単一のト書きから **world_flags 条件つきの段階表**
  （arc_stages）にするスキーマ拡張で受け止められる見込み。
  状態機械は既に world_flags があるので、**新しいランタイム機構は不要**、
  データ形式の拡張で足りるのではないか【要議論】。

### F. Bストーリー照射型（サブプロットによるテーマの照射）

- **設置**: メインの依頼と独立に進行する第二の関係性・小さな物語
  （例：AI冒険者の個人的な過去、村の脇役の小さな悩み）
- **本体**: BストーリーがAストーリーの解決の鍵・視点を与えるという接続（③層）
- **回収**: Break into Three 相当の瞬間に、Bストーリー由来の気づきを注入する
- TRPGでの自然な担い手は **AI冒険者**である。AI_DESIGN 2節は既に AI冒険者の②層に
  「隠し事（仲間に言えない過去等）」を認めており、これはまさにBストーリーの器。
  ソロプレイでは同行NPCか依頼人が担う。
- 問題は制約T1が最も強く効くこと：Bストーリーは**道中の会話**で進行するが、
  プレイヤーが雑談に乗らなければ進行しない。映画は尺で保証するが、卓では保証できない。
- **受け止め案**: Bストーリーの各段階を「章内のどこかで1回発話する」という
  緩い期限の②層指示（「このシーン」ではなく「この章のうちに」）として持つ。
  現行の期限概念（シーン到達）を**章単位の発話ノルマ**に緩めた変種であり、
  台帳スキーマの期限フィールドの型を広げれば載る。
  ただし「会話の自然な流れで差し込む」品質は②層文法の未解決課題
  （AI_DESIGN 6節）と同じ問題圏にある。

### G. トーンの前兆（atmospheric foreshadowing / Whiff of Death）

- **設置**: 事件の前に、天候・音・空気の質感で来るべきものを予感させる
- **本体**: 予感の対象となる事件（③層またはシーン定義の enemy/イベント）
- **回収**: 事件そのもの。回収の演出は不要（事件が起きた時点で前兆は自動的に成就する）
- 実は**現行機構でほぼ扱えている**：②層演出指示の本来の用途（「効果を書く」）そのもの。
  F3 の設置「所作のぎこちなさ・坑道の軋み」は既にトーンの前兆である。
- 台帳に載せる価値は「前兆の強度管理」にある：同じ前兆を複数シーンで**漸増**させる
  （1章「かすかな軋み」→2章「粉塵が落ちる」→3章「崩落」）には、
  シーンをまたいだ強度パラメータが要る。単発なら現機構、漸増させるなら
  台帳エントリに強度段階を持たせる小拡張。

### H. 反復モチーフ / Rule of Three

- **設置**: 同じ意匠・仕草・言い回しを2回、普通のものとして見せる
- **本体**: 3回目で意味が変わる（笑いが哀しみに、日常が喪失に）という設計
- **回収**: 3回目の提示。文脈の変化が意味を変える
- C-1（脚本内呼応）の一般形。台詞に限らず**描写のモチーフ**を含む。
- 機構上の困難は**回数の保証**（制約T1）：プレイヤーが2回目の現場を飛ばすと
  3回目が「初出」になり効果が消える。反復系は「n回目まで提示済みか」という
  **カウンタ状態**を要求する——現行台帳の3状態（planted/revealed/paid_off）では
  表現できない。planted を `presented_count` に一般化するスキーマ変更が要る。
- 強制注入との相性は良い：「2回目が未提示のままシーンXに到達したら、
  移動描写に混ぜて提示せよ」という**追いつき注入**が定義できる。

### I. レッドヘリング（偽の伏線）

- **設置**: 本物の伏線と同じ文法で、誤った推測へ誘導する匂わせを置く
- **本体**: 「それは無関係である」という事実（③層）
- **回収**: 誤誘導の解消。**回収しないと未回収の伏線と区別がつかず、不誠実になる**
- 三層知識モデルとの相性が非常に良い：LLMは③層を知らないので、
  ②層に「作業札の血痕に注意を引け」と書けば、**LLMは本物の伏線と同じ真剣さで
  偽の伏線を演出する**。漏洩リスクゼロの誤誘導。
- 台帳にそのまま載る（種別に「偽装」を追加するだけ）。ただし運用上の論点：
  - ソロ・短編（30分×3章）でレッドヘリングを張る余白があるか
  - プレイヤーが偽の線を深追いした時、GMは付き合うのか切り上げるのか
    （迷走時の再提示＝quest目標の②層注入、との綱引き）【要議論】
- 【要議論】そもそもMVPスコープで扱うか。優先度は低い、が台帳スキーマ設計時に
  種別欄の値として予約だけしておく価値はある。

### J. 予言・約束型（明示的予告）

- **設置**: ①層で**堂々と**予告する（「心石が尽きれば、坑道は主を失う」）。隠さない
- **本体**: 予告の成就条件とその演出（③層…だが半分は公開情報）
- **回収**: 成就（または意図的な裏切り＝サブバージョン）
- 伏線というより**サスペンス装置**（ヒッチコックの爆弾理論：観客が知っているから怖い）。
- 機構的には最も簡単：設置は普通の開示、回収は既存の強制注入。台帳にそのまま載る。
  現行台帳が扱っていないのは単に「隠す伏線」に意識が向いていたため。
  **「隠さない伏線」も台帳の管轄である**、という認識の更新だけで扱える。

## 3. 判定マトリクス —— 既存機構で受け止められるか

| 型 | 設置の担い手 | 回収の担い手 | 制約T1耐性 | 判定 |
|---|---|---|---|---|
| A. モノ・情報型 | ②層＋secrets | 強制注入 | ◎（実証済み） | **現機構で完結** |
| B. 意味の反転型 | 開示済み①層 | 強制注入＋対比再提示 | ◎ | 現機構＋**小拡張**（一次意味/真意味の二段化、反転用②層文法） |
| C-1. 脚本内呼応 | 指定台詞の発話指示 | 変奏指示の注入 | ○（章単位期限で担保） | 現機構＋**小拡張**（設置を「台詞指定」に拡大、変奏の②層文法） |
| C-2. 動的callback | **プレイ中の発話ログ** | 引用指示の注入 | △ | **新機構**（動的伏線台帳。4.1） |
| D. ブックエンド | 章のopening_image | final_imageの②層 | ◎（終幕は必ず来る） | **新スキーマ**だが機構は既存（章定義の拡張。4.2） |
| E. アーク型 | キャスト②層の段階表 | world_flags条件の段階遷移 | ○ | 現機構＋**スキーマ拡張**（arc_stages。4.3） |
| F. Bストーリー | 章単位の発話ノルマ | 気づきの注入 | △（会話依存） | 現機構＋**期限概念の緩和**（シーン到達→章内ノルマ） |
| G. トーンの前兆 | ②層（現行どおり） | 事件の発生（自動） | ◎ | **現機構で完結**（漸増させるなら強度段階の小拡張） |
| H. 反復モチーフ | 提示カウンタつき台帳 | n回目の提示＋追いつき注入 | △（回数保証が課題） | 現機構＋**状態モデル拡張**（カウンタ化） |
| I. レッドヘリング | ②層（本物と同文法） | 解消の開示 | ○ | **現機構で完結**（種別追加のみ。優先度低） |
| J. 予言・約束型 | ①層で公開 | 強制注入 | ◎ | **現機構で完結**（認識の更新のみ） |

まとめると：

- **そのまま載る**: A・G・I・J（4型）。台帳の種別欄と運用ルールの追記だけ
- **スキーマ拡張で載る**: B・C-1・E・F・H（5型）。ランタイム機構は既存のまま、
  台帳・キャスト・期限のデータ形式を広げる
- **新機構が要る**: C-2（動的callback）と D（ブックエンド、ただし小さい）の2つ

**重要な観察**：新しいランタイム機構がほぼ要らないのは偶然ではない。
「設置=効果を②層に書く／本体=③層に隠す／回収=チェックポイントで注入を保証」という
三点セットは、脚本術の plant/payoff の**タイミング制御をチェックポイント制御に
翻訳する汎用フレーム**として設計されており、大半の技法はデータ形式の問題に還元される。
例外は「プレイ中に生まれる素材」（C-2）だけであり、これは映画に存在しない
TRPG固有の技法——**やる価値の順位づけが本書の次の論点になる**。

## 4. 新機構・拡張の候補案（叩き台）

### 4.1 動的伏線台帳（C-2用）——「エンジンが自分のChronicleを読む」

制作時に書かれた静的台帳と別に、プレイ中に追記される動的エントリを持つ。

```
動的エントリ = {
  quote: 実際の発話テキスト（GM語りの全文記録から採取）,
  speaker: 発話者,
  context: 発話時の状況（シーン・フラグ）,
  payoff_condition: 再登場させるべき文脈（例: 番人の最期 / 章の終幕）,
  status: captured → paid_off
}
```

- **採取をどうやるか**が核心。案は2つ：
  - a) 山場の直後に小型モデルで「このシーンの引用価値のある一文」を抽出
    （D-011 のモデル使い分け方針と整合。掛け合い用の小型モデルの追加用途）
  - b) 章末の要約処理（AI_DESIGN 6節「長期キャンペーンの状態要約」）に相乗りする。
    追加呼び出しゼロ
- 回収は既存の強制注入の変種：「終幕の語りに、この一文を変奏して織り込め」。
  引用の正確性は D-016 検証（外部AIが全文ログから名台詞を正確に引用）で
  実証済みの能力の内側にある
- リスク：引用の乱発は嫌味になる。1章1回まで等の予算制が要る【要議論】
- 位置づけ：**Chronicle（事後の資産化）のリアルタイム版**。D-016/D-020 の投資が
  そのままプレイ体験の品質に還流する、筋の良い接続に見える

### 4.2 章スキーマへの opening_image / final_image（D用）

```
chapter に追加:
  opening_image: { brief: 冒頭の情景, motifs: [固定するモチーフ] }
  final_image:   { direction: "opening_imageを引用し、world_flagsの差分を対比して描け" }
```

- 伏線台帳とは独立の**章の額縁**。台帳に無理に載せない方が概念が濁らない
- 終幕シーンは必ず到達するので回収保証の仕掛けが不要＝実装が最も軽い
- キャンペーン全体のブックエンド（1章sc1の坑口 ⇔ 3章終幕の坑口）は
  『ランタンヒル年代記』の現行シーン構成で**今すぐ手動検証できる**
  （F1〜F3 と同じく、モックに手動埋め込みして体感を見る）

### 4.3 キャスト②層の段階表 arc_stages（E用）

```
キャストの②層を単一ト書きから配列に:
  arc_stages:
    - { when: 常時（1章）, direction: "値踏みする目。人を損得で測る所作を挟め" }
    - { when: learned_heartstone, direction: "関心を抑える演技に綻びを見せろ" }
    - { when: heartstone_choice=砕いた, direction: "初めて損得の外の表情を見せろ" }
```

- ランタイムは「world_flags で該当する段階の②層を注入する」だけ。
  報告シーンの分岐（CAMPAIGN_01 3.1）で既にやっていることの一般化
- アークの「同じ状況で違う行動」原則から、各段階は**初期段階と対になる所作**を
  書くのが②層文法のコツになりそう（→ AI_DESIGN 6節「②層の文法」の議題に合流）

### 4.4 台帳スキーマの共通拡張（B・C-1・F・H用）

- 種別欄の拡張: `装飾 / 必須` に加え `反転 / 呼応 / 偽装 / 反復` を予約
- 期限の型の拡張: `シーン到達` に加え `章内（発話ノルマ）` / `終幕`
- 状態の拡張: `planted` を `presented_count: n` に一般化（反復型のため）
- 反転型の二段化: `first_meaning`（開示済み一次意味）＋ 本体（真意味）

## 5. 優先順位の私案（決定ではなく議論の出発点）

体感検証のコスト（モックへの手動埋め込みで済むか）と、物語効果の大きさで並べると：

1. **D. ブックエンド** —— 機構最小・回収保証が構造的に自明・効果は「選択の総量の可視化」
   という本作の核（構造は収束、語りは発散）に直結。第3章を書く時に手動で試せる
2. **J. 予言型 ＋ G. 前兆の漸増** —— 台帳の運用拡張のみ。F3（崩落）は既に
   前兆→予言→成就の三段に組み替えられる素材
3. **E. アーク型（マイラで1本）** —— スキーマ拡張はデータ形式のみ。
   マイラの③層（真の目的）と報告シーン分岐が既にあるので素材が揃っている
4. **C-1. 脚本内呼応** —— キャンペーンの問い「灯を守るのは、誰の仕事か」を
   Theme Stated として1章の台詞に埋める。変奏指示の②層文法の実験を兼ねる
5. **C-2. 動的callback** —— 新機構だが、Chronicle資産との接続が筋良く、
   「AIならでは」の差別化になる。章末要約への相乗り（4.1-b）なら追加コスト小
6. **H. 反復モチーフ** —— カウンタ化は台帳エンジン実装（未着手・製品段階）と同時でよい
7. **I. レッドヘリング** —— MVPスコープ外の判断でよい。種別の予約のみ
8. **F. Bストーリー** —— AI冒険者実装（検証順序②）と不可分。ソロ段階では保留
