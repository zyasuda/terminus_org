# キャラ掛け合い(ボケ/ツッコミ)データ仕様

「廃坑の灯」の同行AI同士・AIと人間の**関係のリズム**をデータ化する試作。
笑いの核心は「間」と「抑制」にある——だから頻度制御はシステムが持つ(相棒ガレスの
発言頻度をシステムが管理したのと同じ思想。[自宅作業.md](自宅作業.md) 課題7参照)。

---

## 0. 設計の3原則(この卓で確認したこと)

1. **ボケは単体では成立しない。誰かのツッコミ(または黙殺)があって初めて成立する。**
   → データは「キャラ単体の癖」と「ペアの反応」の二層で持つ。
2. **時差があってよい。** ボケた直後でなく、ツッコミ役の次の手番で返しても成立する。
   → システムは「ツッコミ権」を発行し、相手が次に喋る/動くまで保持する(pending)。
3. **やりすぎは騒音。4回に1回くらいがちょうどいい。** 頻度はシステムが数える。
   → LLMは「演じる」だけ。「いつ演るか」はシステムが決める(②層＝演出指示に乗せる)。

## 3. 動作モデル(システムは「いつ」、LLMは「何を」)

**役割分担が核心**(2026-07-04プレイの反省から確定):
- **システム=いつ言うか**(頻度・間・トーン抑制)。放っておくとLLMは喋りすぎる/暴発する。
- **LLM=何を言うか**(台詞そのもの)。定型文より文脈に合った生きた台詞を書ける。

処理の流れ:

1. **ボケ検知**: LLMが `companion.aside=true`(頼まれていない癖の一言)を返し、かつ
   それがプレイヤーへの呼びかけ回答でない時だけ、`to` を狙うペアの `banterCharge` を +1。
   → **誠実な回答・進言・警告はボケではない**(「発言=ボケ」と数えると暴発する)。
2. **ツッコミ権の発行**: `banterCharge >= 実retortEvery` で `from` にツッコミ権を1枚発行し
   0 に戻す。実 retortEvery = `基準値 - (from.retortDrive - 3)`(下限2)。ボケの文面
   (`bokeLine`)も一緒に持ち越す——ツッコミが具体的なボケに紐づくように。
3. **時差の許容(原則2)**: ツッコミ権は次の手番の頭で消化。その手番の systemPrompt に
   **「掛け合い許可(この手番のみ・任意)」**という②層指示を注入する。指示には
   ボケの文面と、トーン見本(`tsukkomi[]` の一部)を添える。
4. **中身はLLMが書く**: LLMは見本を**そのまま使わず**、その場に合わせて台詞を書く。
   呆れ・皮肉・「ふん」の黙殺、いずれもよい。**場に合わなければ入れず null でよい**
   ——この裁量が「間」を生む(合わない場面に無理に押し込まない)。
5. **トーンゲート(noBanter)**: 悲劇・厳粛なシーンでは、溜めもflushも抑制し、
   溜まっていたツッコミ権も破棄する。

> 旧版は `tsukkomi[]` から定型文を1つ選んで差し込んでいた。だがボケの無い場面や
> 山場に暴発した(T8/T21)。上記は「システムは許可、LLMが執筆」への転換の結果。

## 4. 既存アーキテクチャへの接続

- **②層(演出指示)に乗る**: 掛け合い許可は「説明しない演出」。systemPrompt が
  その手番だけ注入し、次手番には消える。三層知識モデルにそのまま収まる。
- **頻度・トーンの決定権はシステム側**: ガレスのクールダウンと同じ思想。
  LLMは「いつ」を決めない。「何を」だけを書く。
- **戦闘A案との噛み合わせ**: 「速いガレスが指示を待たず動く」=ボケの主要な発火源。
  制約が笑いの燃料になる。ツッコミは時差OKなので、リディアの手番が後から来ても成立。

## 5. 未検証・要プレイテスト

- `retortEvery` 4 / こだわり補正の体感が妥当か(「間」は乱数＋パラメータで近似するが、
  効くかどうかは実プレイの呼吸に依存する)。
- 黙殺(ignore)の抽選確率をいくつにするか(高すぎると寂しい、低すぎるとくどい)。
- 三人目以降が増えたとき、ペアが N×(N-1) に増える。全ペアを書くか、
  「基本の相性テーブル＋例外」に切り替えるかは、人数が増えてから判断。
