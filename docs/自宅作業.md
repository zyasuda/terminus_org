2026-07-04 セッション内でプレイテストを通じて見つかった課題を記録。
各問題は「原因→修正→検証」の流れで解決。

## 1. 懐中電灯リーク（語彙規範の破裂）

**現象**  
T10 第1回プレイスルー中、ナレーションで「懐中電灯ではなく、松明だ」と出力。
プレイヤーの所持品はランタンのみで、懐中電灯は世界に存在しない。

**根本原因**  
- 世界観の語彙レジスタが `systemPrompt()` にハードコード（自由度が高すぎた）
- エンティティ契約がなく、別名への言い換えをLLMが許可していた
- 修正：実名 1 個の決定論的規則がなかった

**修正**  
- [CAMPAIGN_01.md](CAMPAIGN_01.md) に「語彙レジスタ」セクションを作成（禁止語/必須語の明確化）
- systemPrompt() に明示：
  ```
  シーンに名のある事物(敵・深さのある対象・入手品・所持品)は、
  その名称を一字一句そのまま使え。別の類似物に言い換えるな
  (例:ランタン→松明は不可)
  ```
- エンティティ契約(D-021)を正名の原則として文書化

**検証**  
第2回プレイスルーで同問題は再発しなかった。

## 2. 所持品の無断描写（状態管理の越権）

**現象**  
s1a開示時、ナレーションで「ランタンの灯を掲げ、油はまだ半分…」と、
プレイヤーが宣言していない所持品の内容を勝手に描写。

**根本原因**  
- 「所持品」はシステム管理の状態なのに、LLMが反応的参照ではなく能動的に描写していた
- プロンプトに境界線がなかった

**修正**  
systemPrompt() に明記：
```
所持品は反応的参照のみ。あなたから所持品を描写・言及してはならない。
```

**検証**  
その後のプレイでは所持品は「プレイヤーが示唆したとき」のみ応答に出現。

## 3. ルートのアイテム虚実乖離

**現象**  
T18 で、プレイヤーが「心石の欠片をポケットに入れた」と描写し、
LLMのナレーションでも「受け取った」と語られたが、
ゲームシステムの `state_updates.add_items` は空のままで、戦績欄に記録されず。

**根本原因**  
- `add_items` リクエストが `applyUpdates()` で検証され、
  シーン定義に `loot: ["心石の欠片"]` がなかったため却下された
- LLMとシステムの同期がなく、ナレーション上では「取得」、
  実際のゲーム状態では「所持していない」という矛盾が起きた

**修正**  
- [CAMPAIGN_01.md](CAMPAIGN_01.md) の各シーンに `loot: [...]` 配列を事前定義
  - シーン3 に `"心石の欠片"` を追加
- systemPrompt() に明示：
  ```
  入手品のホワイトリスト外のものはプレイヤーに取得させるな。
  この場面で取得可能なのはリストのものだけ。
  ```

**検証**  
第2回以降、add_items は常にホワイトリスト内のみ。
ナレーション上の「取得」とシステム状態が一致。

## 4. 秘密の表層なし崩し開示

**現象**  
T4 で軌道調査の判定を要求せず、ナレーションで「最近こすれた跡がある」と暴露。
これは s1b の秘密内容そのもの（部品の損傷痕跡＝最近の修復作業の証拠）。

**根本原因**  
- 各 `secrets` には概要は定義されていたが、
  「判定なしに何ここまで語ってよいか」の上限がなかった
- 秘密のレイヤー構造が不明確だった

**修正**  
- 各 `secrets` に `surface: "..."` フィールドを追加
- systemPrompt() で明記：
  ```
  秘密は「()括弧内の表層の範囲まで」のみ開示可。
  括弧を超える内容・正体を判定なしに語ってはならない。
  ```
- 例：
  ```javascript
  {
    id: 's1b',
    trigger: '軌道調査の判定成功',
    surface: '(部品に古い損傷跡。ただし最近こすられた様子はない)',
    content: '修復は〇年前。最近の改変はない。'
  }
  ```

**検証**  
第2回でも、秘密開示は表層の枠内に収まった。

## 5. API通信エラー（インフラ層）

**現象**  
- Gemini free tier: 約15ターン後に HTTP 503 / 429（レート制限）
- モデルID形式エラー：`"claude-Haiku-4.5"` は拒否

**原因**  
- Gemini無料版: 20req/min 制限
- モデルIDは小文字・ダッシュ表記が必須（`claude-haiku-4-5-20251001`）

**修正**  
- `server.js` をAnthropicバックエンド優先に変更（`.env` で LLM_BACKEND 選択可能）
- `.env.example` に正しいモデルID形式を記載
- ユーザーは `/model` で即座に切り替え可能に

**検証**  
Anthropic Haiku 4.5 → Opus 4.8 へシームレス切り替え。問題なし。

## 6. サーバープロセスの不意の死

**現象**  
プレイスルー中に「Failed to fetch」エラー、サーバープロセス沈黙。

**根本原因**  
`server.js` に例外ハンドラがなく、
リクエスト処理中のエラーでプロセスが意図的に終了していた。

**修正**  
`server.js` 末尾に追加：
```javascript
process.on("uncaughtException", e => console.error("uncaughtException:", e));
process.on("unhandledRejection", e => console.error("unhandledRejection:", e));
```
ログに記録しながら動作継続。

**検証**  
その後のプレイで、エラー発生時も サーバーは生存。
ブラウザ側で通信エラーとしてユーザーに報告される。

## 7. ガレスの冷却時間と会話の最低限ルール

**現象**  
T14 第2回プレイスルー中、プレイヤーが「大丈夫か」と気遣う
（ガレスに直接話しかけ）→ ガレスは沈黙。
クールダウン（3ターン）がまだ有効で、返答を拒否していた。

**根本原因**  
- `maybeCompanion()` が強制的に3ターンクールダウンを適用
- 会話の基本ルール（話しかけられたら返す）が周波数制御に踏み潰された

**修正**  
- `maybeCompanion(r, addressed)` にフラグを追加
- 呼び出し側（`sendAction()`）でプレイヤー宣言に「ガレス」が含まれるかを判定
- `addressed=true` ならクールダウンをバイパス、必ず応答
- systemPrompt() に追加：
  ```
  プレイヤーがガレスに直接話しかけた・尋ねた・気遣った時は、
  必ず何か返す(黙殺は不可。分からない事は「分からん」でよい)
  ```

**検証**  
同じ状況（クールダウン中に「ガレス、〇〇か？」）を再現 → 応答あり。
ガレスが「心配をかけた。だが大事はない。それより——あの音だ。確かめに行くか？」と返答。

**設計原則**  
「相棒の会話の最低限のルール（話しかけられたら返す）は、
周波数制御より優先される」

## 修正による副次効果

- **エンティティ契約の定式化**: 実名の原則により、
  LLMが新しい呼び方を創造することがなくなった
- **表層概念の浸透**: その後、秘密の開示が段階的になり、
  判定チェックの緊迫感が増した
- **ガレス会話の活性化**: 呼びかけ応答により、
  プレイヤーとNPCの関係性が劇的に深まった
